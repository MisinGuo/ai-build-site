# 全行业建站工具升级方案

> 基于现有 game-box 系统演进，升级为 **AI 驱动的全行业通用建站平台**
> 版本：v3.0 | 日期：2026-04-06

---

## 一、核心设计思路

### 1.1 要解决的根本问题

现有系统每扩展一个行业，必须同步修改：

- 后端：新写 Controller / Service / Mapper / 数据库表
- 前端管理后台：新写管理页面
- 展示端：新写 Next.js 页面、新加 API 类型定义

这意味着**任何新行业都依赖程序员介入**，AI Agent 无法独立完成建站。

### 1.2 目标

```
用户（或 AI Agent）只需要：
  1. 定义"这类内容有哪些字段"（Schema）
  2. 录入数据
  3. 配置站点页面结构（site_config）

后端、前端展示端：0 行代码改动。
```

### 1.3 核心范式

不让 AI 生成前端代码，而是让 **AI 生成驱动前端的配置（site_config）**。

```
旧方式：  用户需求  程序员写代码  行业专属页面上线
新方式：  用户需求  AI 生成 site_config JSON  统一渲染器解释  站点上线
```

## 二、整体架构

```

                      管理平面 (Admin)                                 
                                                                      
        
     内容管理              AI 建站助手           站点配置管理      
    - 内容类型定义        （AI 对话           - site_config    
    - 内容项录入           生成配置 JSON）      - 版本历史        
    - 关联关系管理          
                                                  

         Content API（统一，行业无关）     site_config JSON

                   展示平面 (Display)                                  
                                                                      
   Universal Renderer（一套 Next.js，通过 site_config 服务所有站点）    
   - 根据 site_config.pages 决定有哪些路由                             
   - 根据 site_config.theme 决定视觉风格                               
   - 根据 section.component 决定用哪个区块组件                          
   - 区块组件从 Content API 拉数据，通过 schema 动态渲染字段             
                                                                      

```

**关键原则：Universal Renderer 只部署一次，所有站点共用。通过 Host 头识别站点，拉取各自 site_config + 内容数据。**

## 三、数据库设计（后端统一核心）

### 3.1 四张核心通用表

这四张表是整个统一后端的基础，**建完之后不论新增多少行业都不再加表**。

#### 表一：内容类型定义

```sql
CREATE TABLE gb_content_types (
  id          BIGINT PRIMARY KEY AUTO_INCREMENT,
  site_id     BIGINT,                    -- NULL 表示全局模板类型
  code        VARCHAR(50) NOT NULL,      -- 类型编码：house / product / course / doctor ...
  name        VARCHAR(100) NOT NULL,     -- 显示名称
  schema_json MEDIUMTEXT NOT NULL,       -- 字段定义（FieldSchema[]），见下方说明
  status      CHAR(1) DEFAULT '1',
  sort_order  INT DEFAULT 0,
  create_time DATETIME,
  update_time DATETIME,
  INDEX idx_site_code (site_id, code)
);
```

`schema_json` 格式（定义"房源"类型示例）：
```json
[
  { "key": "price",       "type": "number",     "label": "价格(万元)", "required": true },
  { "key": "area",        "type": "number",     "label": "面积(㎡)",   "required": true },
  { "key": "rooms",       "type": "select",     "label": "户型",       "options": ["1室","2室","3室","4室+"] },
  { "key": "district",    "type": "text",       "label": "区域",       "required": true },
  { "key": "images",      "type": "image_list", "label": "图集" },
  { "key": "description", "type": "richtext",   "label": "详情" }
]
```

支持的字段类型：`text` / `textarea` / `richtext` / `number` / `image` / `image_list` / `select` / `date` / `url` / `tags`

#### 表二：通用内容项

```sql
CREATE TABLE gb_content_items (
  id           BIGINT PRIMARY KEY AUTO_INCREMENT,
  site_id      BIGINT NOT NULL,

  title        VARCHAR(500),           -- 主标题（搜索/SEO 用）
  slug         VARCHAR(255),           -- URL 友好标识
  cover_image  VARCHAR(500),           -- 封面图
  fields_json  MEDIUMTEXT,             -- 按 schema 填写的动态字段
  seo_title    VARCHAR(200),
  seo_desc     VARCHAR(500),
  status       CHAR(1) DEFAULT '1',
  sort_order   INT DEFAULT 0,
  view_count   INT DEFAULT 0,
  create_time  DATETIME,
  update_time  DATETIME,
  INDEX idx_site_type   (site_id, type_code),
  INDEX idx_slug        (site_id, slug),
  INDEX idx_sort        (site_id, type_code, sort_order)
);
```

#### 表三：内容项关联关系

替代所有行业专属的关联表（`gb_box_game_relations`、`gb_master_article_games` 等）。

```sql
CREATE TABLE gb_content_relations (
  id            BIGINT PRIMARY KEY AUTO_INCREMENT,
  from_id       BIGINT NOT NULL,       -- 来源内容项 id（如楼盘 id）
  to_id         BIGINT NOT NULL,       -- 目标内容项 id（如房源 id）
  relation_type VARCHAR(100) NOT NULL, -- 关联类型：project_has_unit / article_ref_item / gamebox_has_game ...
  sort_order    INT DEFAULT 0,
  meta_json     TEXT,                  -- 关联元数据（如折扣、楼层等行业专属信息）
  create_time   DATETIME,
  INDEX idx_from (from_id, relation_type),
  INDEX idx_to   (to_id,   relation_type),
  UNIQUE KEY uk_from_to_type (from_id, to_id, relation_type)
);
```

使用示例：
```
relation_type = "gamebox_has_game"
meta_json = {"discountLabel":"7折","firstChargeDomestic":0.7,"hasSupport":true}

relation_type = "project_has_unit"
meta_json = {"buildingNo":"A栋","floor":"3F"}

relation_type = "article_ref_item"
meta_json = null
```

#### 表四：站点内容可见性控制

替代所有行业专属的站点关联表（`gb_site_game_relations`、`gb_site_box_relations` 等）。

```sql
CREATE TABLE gb_site_content_access (
  id           BIGINT PRIMARY KEY AUTO_INCREMENT,
  site_id      BIGINT NOT NULL,
  item_id      BIGINT NOT NULL,
  access_type  VARCHAR(20) NOT NULL,  -- include（显式包含）/ exclude（显式排除）
  is_visible   CHAR(1) DEFAULT '1',
  is_featured  CHAR(1) DEFAULT '0',   -- 是否精选/置顶
  custom_name  VARCHAR(200),          -- 该站点下的自定义名称（可覆盖 item.title）
  sort_order   INT DEFAULT 0,
  meta_json    TEXT,                  -- 站点专属覆盖数据（如自定义下载链接、价格等）
  create_time  DATETIME,
  update_time  DATETIME,
  INDEX idx_site   (site_id, is_visible),
  INDEX idx_item   (item_id),
  UNIQUE KEY uk_site_item (site_id, item_id)
);
```

#### 表五：内容项分类关联

复用现有 `gb_categories` 表，增加一张通用关联表替代行业专属的 `gb_game_category_relations`、`gb_box_category_relations` 等。

```sql
CREATE TABLE gb_content_item_category_relations (
  id          BIGINT PRIMARY KEY AUTO_INCREMENT,
  item_id     BIGINT NOT NULL,        -- 关联 gb_content_items.id
  category_id BIGINT NOT NULL,        -- 关联 gb_categories.id
  is_primary  CHAR(1) DEFAULT '0',    -- 是否主分类（用于列表展示和搜索权重）
  sort_order  INT DEFAULT 0,
  create_time DATETIME,
  INDEX idx_item     (item_id),
  INDEX idx_category (category_id),
  UNIQUE KEY uk_item_category (item_id, category_id)
);
```

**与 `gb_categories` 的关系**：`gb_categories.categoryType` 现有值为 `game/drama/article/gamebox` 等，新增行业直接将 `categoryType` 设为对应的 `typeCode`（如 `house` / `course` / `doctor`），无需修改 `gb_categories` 表结构。一个内容项可以挂多个分类，`is_primary='1'` 的分类用于列表展示和 SEO canonical 分类路径。

分类筛选查询示例：
```sql
-- 查询某分类下的所有房源，按主分类优先排序
SELECT ci.* FROM gb_content_items ci
INNER JOIN gb_content_item_category_relations cr ON ci.id = cr.item_id
INNER JOIN gb_categories c ON cr.category_id = c.id
WHERE ci.site_id = ? AND ci.type_code = 'house'
  AND c.slug = 'pudong'
ORDER BY cr.is_primary DESC, ci.sort_order;
```

### 3.2 为什么是这五张表

```
gb_content_types                      回答"有什么字段"
gb_content_items                      回答"有什么数据"
gb_content_relations                  回答"数据之间怎么关联"
gb_site_content_access                回答"哪个站点能看到哪些数据"
gb_content_item_category_relations    回答"数据属于哪些分类"（支持搜索筛选）
```

这五个问题覆盖了任何行业的数据需求。

### 3.3 多语言支持机制

现有系统中 `gb_translations` 表已实现通用多语言存储（文章系统在用），结构为：

```
gb_translations: entity_type | entity_id | locale | field_name | field_value
```

`gb_content_items` **完全复用这张表**，无需新建任何多语言表。

#### 存储规则

| 字段 | 默认语言 | 其他语言（存入 gb_translations） |
|------|---------|--------------------------------|
| `title` | 存在 `gb_content_items.title` | `field_name='title'` |
| `seo_title` | 存在 `gb_content_items.seo_title` | `field_name='seo_title'` |
| `seo_desc` | 存在 `gb_content_items.seo_desc` | `field_name='seo_desc'` |
| `fields_json` 中可翻译字段（text/textarea/richtext） | 存在 `fields_json` 本身 | `field_name='fields.{key}'`，如 `fields.description` |
| `fields_json` 中非翻译字段（number/image/select） | 存在 `fields_json`，各语言共享 | — |

查询时 `entity_type = 'content_item'`，批量时用现有 `selectBatchTranslations`（按 entity_ids 批量取）。

#### API 响应多语言合并逻辑

```java
// 伪代码：Content API 返回前的多语言合并
List<ContentItem> items = contentItemMapper.selectList(query);
if (locale != null && !locale.equals(defaultLocale)) {
    List<GbTranslation> translations = translationMapper.selectBatchTranslations(
        "content_item", locale, itemIds);
    // 合并：translation 的值覆盖 item 的对应字段
    mergeTranslations(items, translations);
}
```

#### schema 中标记可翻译字段

`schema_json` 中新增 `translatable` 属性，管理后台据此显示多语言输入框：

```json
[
  { "key": "price",       "type": "number",   "label": "价格",   "translatable": false },
  { "key": "description", "type": "richtext", "label": "详情",   "translatable": true },
  { "key": "district",    "type": "text",     "label": "区域",   "translatable": true }
]
```

**`translatable: false` 的字段（价格、图片、数字）在所有语言下共享同一个值，不进翻译表。**

#### 分类名称的多语言

`gb_categories.name` 通过 `gb_translations` 翻译：`entity_type='category'`, `field_name='name'`。分类多语言与现有游戏分类完全一致，无需额外工作。

#### 搜索接口的多语言参数

`/api/public/content-items?typeCode=house&locale=en` 时，API 先查内容项，再批量取翻译，合并后返回。`categorySlug` 过滤同样支持多语言分类名搜索（通过 translations 表关联）。

### 3.4 与现有游戏/短剧系统的关系

## 四、统一 Content API 设计

所有端点**不再有行业词**，永远不需要新加接口。

```
GET  /api/public/site-info
      返回站点基础信息 + site_config JSON（渲染器启动时调用）

GET  /api/public/content-items
     ?typeCode=house&siteId=1&locale=zh-CN&categorySlug=pudong&page=1&size=20
      返回内容项列表，同时携带该类型的 schema

GET  /api/public/content-items/{slug}
     ?typeCode=house&siteId=1&locale=zh-CN
      返回单条内容项详情 + schema + 关联内容

GET  /api/public/articles           （保留，不变）
GET  /api/public/articles/{slug}    （保留，不变）
GET  /api/public/categories         （保留，不变）
GET  /api/public/search             （扩展支持 content_items）

兼容保留（游戏行业旧接口）：
GET  /api/public/games               内部路由到 content-items?typeCode=game
GET  /api/public/boxes               内部路由到 content-items?typeCode=gamebox
```

### API 响应格式（统一）

```json
{
  "code": 200,
  "data": {
    "schema": [
      { "key": "price", "type": "number", "label": "价格(万元)" },
      { "key": "area",  "type": "number", "label": "面积(㎡)" }
    ],
    "items": [
      {
        "id": 1,
        "title": "浦东新区精装3居室",
        "slug": "pudong-3br-001",
        "coverImage": "https://...",
        "fields": { "price": 580, "area": 92, "rooms": "3室2厅" }
      }
    ],
    "total": 48
  }
}
```

**schema 随数据一起返回**，前端永远不需要提前知道这是什么行业。

## 五、前端展示端（Universal Renderer）

### 5.1 核心机制

只有一套 Next.js 代码，通过 `site_config` 驱动所有站点的路由、布局、风格。

```typescript
// 启动：根据域名加载站点配置
async function loadSiteConfig(host: string): Promise<SiteConfig> {
  return fetch(`${API_BASE}/api/public/site-info`, { headers: { 'X-Host': host } })
    .then(r => r.json()).then(r => r.data.siteConfig)
}

// 动态路由：一个文件处理所有路径
// src/app/[locale]/[...slug]/page.tsx
export default async function DynamicPage({ params }) {
  const config = await loadSiteConfig(...)
  const pageConfig = config.pages.find(p => matchPath(p.path, params.slug))
  if (!pageConfig) return notFound()
  return <PageRenderer pageConfig={pageConfig} theme={config.theme} />
}
```

### 5.2 site_config 结构

```typescript
interface SiteConfig {
  siteId: number
  name: string
  domain: string
  defaultLocale: string
  supportedLocales: string[]

  theme: {
    colorPrimary: string    // "#3B82F6"
    colorBg: string         // "#0a0a14"
    colorText: string       // "#ffffff"
    fontFamily: string
    borderRadius: string    // "8px"
    preset?: string         // "dark-tech" | "light-minimal" | "warm-organic"
  }

  pages: Array<{
    path: string            // "/listings" | "/listings/:slug"
    title: string
    sections: Array<{
      component: string     // 区块组件名（白名单内）
      props: {
        contentTypeCode?: string  // 从哪种类型取数据
        limit?: number
        layout?: "grid" | "list" | "masonry"
        [key: string]: any
      }
    }>
  }>

  globalBlocks: {
    header: object
    footer: object
    seo: { titleSuffix: string; defaultDescription: string }
  }

  contentTypes: Array<{
    code: string
    listPath: string        // "/listings"
    detailPath: string      // "/listings/:slug"
  }>
}
```

### 5.4 ContentDetail：行业无关的动态渲染

这是整套方案的核心前端组件。它不知道、也不关心内容是房源还是课程：

```tsx
export function ContentDetail({ item, schema }) {
  return (
    <div>
      <h1>{item.title}</h1>
      {item.coverImage && <img src={item.coverImage} />}
      {schema.map(field => (
        <div key={field.key}>
          <label>{field.label}</label>
          <FieldRenderer type={field.type} value={item.fields[field.key]} />
        </div>
      ))}
    </div>
  )
}

function FieldRenderer({ type, value }) {
  switch (type) {
    case 'number':     return <span>{value?.toLocaleString()}</span>
    case 'text':       return <p>{value}</p>
    case 'richtext':   return <div dangerouslySetInnerHTML={{ __html: value }} />
    case 'image':      return <img src={value} />
    case 'image_list': return <>{value?.map(src => <img key={src} src={src} />)}</>
    case 'url':        return <a href={value}>{value}</a>
    case 'select':     return <span className="badge">{value}</span>
    default:           return <span>{String(value ?? '')}</span>
  }
}
```

## 七、核心机制详解：为什么前后端都不需要改代码

### 7.1 接口不再绑定行业

```
旧方式（每行业一套接口）：
  /api/public/games          游戏专属
  /api/public/houses         房产专属（需要新写）
  /api/public/courses        教育专属（需要新写）

新方式（一个接口覆盖所有）：
  /api/public/content-items?typeCode=game
  /api/public/content-items?typeCode=house
  /api/public/content-items?typeCode=course
   后端只有一个 Controller 方法，永远不需要扩展
```

### 7.2 类型定义不再绑定行业

```typescript
// 旧方式（每行业一套类型）：
interface GameDetailResponse { downloadUrl: string; rating: number; ... }
interface HouseDetailResponse { price: number; area: number; ... }  // 需要新写

// 新方式（一套类型，schema 动态描述字段）：
interface ContentItemResponse {
  schema: FieldSchema[]        // 后端告诉前端"有哪些字段"
  item: {
    title: string
    fields: Record<string, any> // 动态字段，行业无关
  }
}
// 加新行业：TypeScript 文件 0 改动
```

### 7.3 前端路由不再绑定行业

```typescript
// 旧方式（每行业一套文件）：
// src/app/[locale]/games/page.tsx
// src/app/[locale]/houses/page.tsx   需要新建文件

// 新方式（一个 catch-all 路由）：
// src/app/[locale]/[...slug]/page.tsx
// 路由定义在 site_config.pages JSON 里，AI/人工可直接修改
```

### 7.4 新增内容类型的完整流程（0 行代码）

```
以"医生"内容类型为例：

步骤1：管理后台  内容类型管理  新建
  typeCode: "doctor"
  schema: [
    { key: "specialty", type: "text",   label: "专科" },
    { key: "hospital",  type: "text",   label: "所在医院" },
    { key: "photo",     type: "image",  label: "医生照片" },

步骤2：管理后台  AI 建站助手  输入

  AI 修改 site_config.pages，追加两项

步骤3：管理后台  内容项管理  选择"医生"  录入数据
  （或通过 AI 工作流批量生成/导入）

结果：
  - 前端立即可访问 /doctors、/doctors/zhang-san
  - ContentDetail 组件根据 schema 渲染所有字段

```

## 八、部署策略

### 主推：多租户单部署

```
Universal Renderer 部署到 Cloudflare Pages（一次）
每个新站点：绑定 Custom Domain  DNS 指向同一个 Pages 项目
渲染器通过 Host 头识别站点  拉取各自 site_config

优势：
- 新站点上线 = 数据库加记录 + DNS 配置，无需 CI/CD
- 渲染器 bug 修复一次，所有站点受益
- 组件库扩充一次，所有站点可用
```

### 降级：独立部署（高度定制场景）

当某站点需要完全自定义的交互逻辑时，可以 fork Universal Renderer 单独部署，这是特殊场景而非默认路径。

## 十一、AI Agent 的操作边界

明确了以上机制后，AI Agent 能独立完成什么：

| 任务 | Agent 操作 | 改代码 |
|------|-----------|--------|
| 新建内容类型（如"医生"）| POST /api/admin/content-types | ❌ |
| 给站点页面加一个新区块 | PATCH site_config | ❌ |
| 批量录入内容数据 | POST /api/admin/content-items（批量）| ❌ |
| 修改站点主题颜色/字体 | PATCH site_config.theme | ❌ |
| 设置内容项的站点可见性 | POST /api/admin/site-content-access | ❌ |
| 新增一种 UI 交互模式（如地图视图）| 向组件库添加 MapView 组件 | ✅ 一次性 |

**结论：数据层和路由层的所有变化都可以由 AI Agent 通过 API 完成，不触碰代码。唯一需要写代码的情况是新增全新的 UI 交互模式。**
