# 原子工具实现方案文档

## 一、设计思路

### 1.1 核心理念
将 `gb_content_tools` 的执行能力抽象化，通过统一的执行器接口（ToolExecutor）实现原子工具的调用。

### 1.2 架构分层
```
Controller层 (GbAtomicToolController)
    ↓ 
Service层 (GbAtomicToolServiceImpl) 
    ↓
ToolExecutor接口
    ↓
具体执行器 (AiGeneratorToolExecutor, TranslatorToolExecutor, etc.)
    ↓
底层服务 (IGbAiToolService, RestTemplate, etc.)
```

## 二、已实现的功能

### 2.1 后端实现

#### ✅ 数据层 (Mapper)
- **GbAtomicToolMapper.java** - Mapper接口
- **GbAtomicToolMapper.xml** - MyBatis映射文件
- 支持CRUD操作和按工具编码查询

#### ✅ 执行器基础设施
- **ToolExecutor.java** - 执行器接口
  - `execute(Map<String, Object> params)` - 执行工具
  - `validateParams()` - 验证参数
  - `getType()` - 获取执行器类型

- **AiGeneratorToolExecutor.java** - AI生成工具执行器
  - 类型：`ai`
  - 功能：调用AI平台生成内容
  - 支持：提示词渲染、参数验证、变量替换

#### ✅ Service层完善
- **GbAtomicToolServiceImpl.java**
  - 自动注册所有ToolExecutor实现
  - 根据工具类型分发执行请求
  - 统一的错误处理和日志记录

#### ✅ Controller层
- **GbAtomicToolController.java**
  - `POST /test/{toolCode}` - 测试工具
  - `POST /execute/{toolCode}` - 执行工具
  - `GET /code/{toolCode}` - 获取工具定义

### 2.2 前端API
- **atomicTool.js**
  - `testAtomicTool(data)` - 测试工具
  - `executeAtomicTool(toolCode, params)` - 执行工具
  - `getToolDefinition(toolCode)` - 获取工具定义

### 2.3 数据库
- **gamebox_v2.89_atomic_tool_data.sql**
  - 文本优化工具 (`text_optimizer`)
  - AI文章生成器 (`ai_article_generator`)
  - SEO关键词提取 (`keyword_extractor`)
  - 文章摘要生成器 (`summary_generator`)

## 四、使用方法

### 4.1 在前端测试工具

```javascript
import { executeAtomicTool } from '@/api/gamebox/atomicTool'

// 调用文本优化工具
const result = await executeAtomicTool('text_optimizer', {
  originalText: '需要优化的文本内容...'
})

// 结果格式
{
  "status": "success",
  "message": "执行成功",
  "output": {
    "generatedText": "优化后的文本...",
    "wordCount": 150
  }
}
```

### 4.2 在文章编辑器中使用

```vue
<template>
  <el-button @click="optimizeText">优化文本</el-button>
</template>

<script setup>
import { executeAtomicTool } from '@/api/gamebox/atomicTool'

async function optimizeText() {
  const selectedText = getEditorSelection() // 获取编辑器选中的文本
  
  if (!selectedText) {
    ElMessage.warning('请先选中文本')
    return
  }
  
  try {
    const res = await executeAtomicTool('text_optimizer', {
      originalText: selectedText
    })
    
    if (res.data.status === 'success') {
      // 替换编辑器中的选中文本
      replaceEditorSelection(res.data.output.generatedText)
      ElMessage.success('文本优化成功')
    } else {
      ElMessage.error(res.data.message)
    }
  } catch (error) {
    ElMessage.error('优化失败: ' + error.message)
  }
}
</script>
```

### 4.3 在工作流中使用

工作流定义示例：
```json
{
  "stepId": "step1",
  "stepName": "优化文本",
  "stepType": "tool",
  "toolCode": "text_optimizer",
  "toolParams": {
    "originalText": "${input.text}"
  },
  "outputMapping": {
    "optimizedText": "${step1.output.generatedText}"
  }
}
```

## 八、常见问题

### Q1: 工具执行失败，提示"工具不存在"
**A:** 检查是否已执行SQL初始化数据，确认 `tool_code` 正确

### Q2: AI工具执行失败，提示"请配置AI平台ID"
**A:** 检查工具的 `config` 字段中是否包含 `aiPlatformId`，确认该AI平台已配置并启用

### Q3: 工具返回结果为空
**A:** 检查 AI平台配置是否正确，API Key是否有效，网络是否正常

### Q4: 如何添加新的工具类型？
**A:** 
1. 创建新的 ToolExecutor 实现类（如 `TranslatorToolExecutor`）
2. 实现 `execute()` 和 `getType()` 方法
3. 添加 `@Component` 注解，Spring会自动注册
4. 在数据库中添加对应类型的工具记录

## 九、总结

本方案成功实现了原子工具的基础架构，主要亮点：

1. **可扩展性**：通过 ToolExecutor 接口，可以轻松添加新的工具类型
2. **自动注册**：Spring自动扫描并注册所有执行器
3. **统一接口**：Controller和Service提供统一的调用接口
4. **复用现有能力**：直接使用 GbContentTool 的 AI调用能力
5. **灵活配置**：工具配置、输入输出定义都通过JSON存储，便于维护

下一步重点是完成文章编辑器和工作流引擎的集成，使原子工具能在实际业务场景中使用。
